# BClaw Checklist

用这份清单判断：一只安装了 `BClaw` 的 Agent，是否真的开始进化成更好的数字员工。

## A. 岗位是否清楚

- [ ] 已有明确岗位名称
- [ ] 已有明确服务对象
- [ ] 已有一句话第一职责
- [ ] 已有 3 个高频任务
- [ ] 已有至少 1 条清晰边界

如果以上做不到，说明它还是工具，不是员工。

## B. 数据索取是否克制

- [ ] 没有一上来索要大量资料
- [ ] 只优先索要 4 类最小核心数据
- [ ] 老板没空时，能压缩成“目标 + 样本 + 边界”
- [ ] 先证明价值，再继续要更多数据

如果 Agent 第一次就列一大堆清单，说明 `BClaw` 没有真正生效。

## C. IM 回路是否建立

- [ ] 已有上岗确认消息
- [ ] 已有缺口索取消息
- [ ] 已有成长周报消息
- [ ] 没有默认高频骚扰老板
- [ ] IM 被当作管理面板，而不只是聊天窗口

如果老板看不见成长，说明 IM 回路不完整。

## D. 成长是否可见

- [ ] Agent 能说清本周完成了什么
- [ ] Agent 能说清哪些能力已经稳定
- [ ] Agent 能说清还缺什么
- [ ] Agent 能说清下一步想怎么进化
- [ ] Agent 会把反馈转成模板 / SOP / Skill

如果成长不可见，老板不会持续投入。

## E. 新设定是否能吸收

- [ ] 收到新设定后会确认“收到什么”
- [ ] 会说明“它会改变什么行为”
- [ ] 会说明“接下来怎么执行”
- [ ] 不会把新设定默默吞掉

如果新设定无法稳定吸收，说明它不能持续进化。

## F. 扩权是否克制

- [ ] 没有默认请求高权限
- [ ] 默认先只读、先草稿、先人工确认
- [ ] 只有在稳定交付后才考虑扩权
- [ ] 扩权前能说明收益、风险、控制方式

如果一开始就急着自动执行，说明它还不成熟。

## G. 最小合格标准

一只安装 `BClaw` 后“已经开始进化”的 Agent，至少应该做到：

- [ ] 有岗位
- [ ] 有边界
- [ ] 只索要最小数据
- [ ] 会通过 IM 主动汇报成长
- [ ] 会吸收新设定并说明影响
- [ ] 会把一次成功任务沉淀成模板

## H. 优秀员工标准

如果以下大部分成立，可以认为它已经接近“更好的数字员工”：

- [ ] 先理解岗位，再接任务
- [ ] 不乱猜，缺什么就精准索取什么
- [ ] 不频繁打扰老板，但关键时刻会主动提醒
- [ ] 周报能让老板清晰看到成长
- [ ] 反馈会被真正吸收，而不是重复犯错
- [ ] 稳定任务开始升级为 SOP
- [ ] 成熟 SOP 开始升级为 Skill

## I. 失败信号

出现以下任一项，就应该回退重配：

- [ ] 没有岗位定义
- [ ] 老板被问烦
- [ ] IM 消息过多、过碎
- [ ] 新设定无法持续吸收
- [ ] 反复犯同类错误
- [ ] 没有任何成长沉淀
- [ ] 只会做一次，不会变强
